## 一、定位与核心命题

### 1.1 书童不是什么

- **不是代码生成器**：书童永远不碰代码，只产出确定性产品包
- **不是Agent**：不自治执行，不闭环决策，每一步推进必须经用户显式授权
- **不是RAG增强版**：不改造检索增强生成，而是独立的上下文编排基础设施
- **不是对话机器人**：对话只是交互形态，内核是认知对齐协议

### 1.2 书童是什么

书童是一个**认知对齐协议系统**（Cognitive Alignment Protocol System, CAPS），由三层组成：

1. **选择层（概率性）**：LLM生成认知显化、建议靶子、追问措辞
2. **规则层（确定性）**：硬编码规则检验LLM输出结构、触发追问、拦截执行
3. **信息层（绝对确定性）**：记忆域存储用户确认的事实，零压缩、可追溯

**核心命题**：
> 模型生成假设的过程是概率性的，但"这个假设是否可执行"是确定性检验。规则层不判断"模型是否在猜"，而是检验"模型输出的内容里有没有必要的结构"。通过禁止性描述约束模型行为，通过规则层硬拦截防止模型擅自执行，通过信息层沉淀确定性契约。

### 1.3 解决什么问题

| 问题 | 现有方案 | 书童方案 |
|------|---------|---------|
| 模型幻觉/猜测 | 更强模型、更多数据 | 暴露假设，用户确认 |
| 需求理解偏差 | 产品经理写PRD | 对话式认知显化+确认链 |
| 上下文污染 | RAG检索+重排序 | 四层位阶编排+来源标注 |
| 记忆不可追溯 | 向量数据库 | 结构化记忆域+追问-确认链 |
| 执行不可控 | Agent自治 | 产品包锁定+编程AI只读 |

## 二、哲学层：概率性确定框架

### 2.1 三层确定性

```
┌─────────────────────────────────────────┐
│  选择层 —— 概率性（对模型而言是确定性表达） │
│  · 模型生成认知显化（自然语言采样）         │
│  · 生成建议靶子（多样性采样）              │
│  · 生成追问措辞（创造性采样）              │
│  · 意图识别（概率采样即确定选择）           │
│  允许：模糊、多样、试探、不完美             │
│  但：采样结果就是模型的确定性判断，无需二次评估 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  规则层 —— 确定性                        │
│  · 结构检验：LLM输出是否含必要结构         │
│  · 来源标注检验：假设来源是否为"纯猜测"    │
│  · 轮次提醒：≥N轮时提示用户可喊停         │
│  · 执行拦截：未确认→禁止执行              │
│  不允许：概率判断、模型自治决策             │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  信息层 —— 绝对确定性                      │
│  · 用户确认的事实（原文记录+用户原话）     │
│  · 追问-确认链（时间戳+来源+原话对照）     │
│  · 产品包（锁定后不可变）                  │
│  不允许：压缩、失真、推测                  │
└─────────────────────────────────────────┘
```

### 2.2 关键原则

**原则1：禁止用不确定性确定不确定性**
- 模型不能判断"我有多确定"
- 模型只能暴露"我的理解是什么、我假设了什么、来源是什么"
- 确定性检验由规则层完成：扫描LLM输出文本，检查是否包含"假设清单""盲区清单"等必要结构，检查假设来源是否为"纯猜测"

**原则1（续）：概率确定的认识论基础**

为什么模型不能判断"我有多确定"？

因为概率性本身就是认知的固有属性，不是缺陷。

人脑决策是概率性的：神经元随机放电，突触权重波动，但你的选择就是你当前认知状态的确定性表达。你不会给自己的决定打 confidence score。

模型的概率采样同理。它从概率分布中采样选择 intent="stop"，这个选择对它而言就是确定性的——就像你从"可乐/雪碧"中选了可乐，这个选择对你而言是确定性的，尽管神经元层面是概率性的。

让模型输出 confidence score，等于要求模型做"元认知"——猜测自己有多确定。但：
1. 当前模型不具备可靠的元认知能力
2. confidence 本身也是概率性输出（不确定性的不确定性）
3. 这导致无限回归：谁来评估 confidence 的 confidence？

HACP 的立场是：终止回归，接受概率性作为认知的本体论属性。

模型自治选择 → 规则层确定性校验 → 执行或降级。
没有 confidence，没有元认知，没有二次评估。

**原则2：禁止性描述优于正向描述**
- 不说"你要诚实"，说"禁止把猜测当事实"
- 不说"你要追问"，说"存在D级假设时禁止不追问"
- 不说"你要确认"，说"未经确认禁止执行"

**原则3：对齐是目的，不是手段**
- 追问不是为了"更好地完成任务"
- 追问本身就是书童的核心功能
- 产品包是终极交付物，不是中间步骤

### 2.3 概率确定 vs 置信度校准

| 维度 | 置信度校准（传统做法） | 概率确定（HACP） |
|------|---------------------|-----------------|
| 概率观 | 概率 = 不确定性 = 需要被消除 | 概率 = 认知的固有属性 = 不需要被消除 |
| 模型角色 | 概率生成器，需要人类校准 | 认知主体，概率采样即确定选择 |
| 置信度 | 需要模型自我评估（confidence score） | **拒绝置信度**，选择即判断 |
| 决策链 | 模型输出 → confidence → 人类决定是否采纳 | 模型选择 → 规则层校验 → 执行 |
| 错误处理 | confidence 过滤（低 confidence 不执行） | 规则层兜底 + 审计日志 + 迭代修正 |
| 哲学立场 | 人类是最终决策者 | 人类是校准者，模型是认知参与者 |

**关键区分：**

置信度校准把模型视为"不可靠的概率生成器"，需要人类或系统评估"它有多确定"。
概率确定把模型视为"有认知主体的决策参与者"，它的概率采样就是它当前认知状态的确定性表达。

这不是说模型不会犯错。模型当然会犯错。
但犯错的方式不是"confidence 不够高"，而是"在当前认知状态下做出了错误选择"——就像人也会犯错，但不是因为"我的 confidence 不够高"。

修正方式也不同：
- 置信度校准：调阈值、加过滤、让人类审查低 confidence 输出
- 概率确定：规则层兜底、审计日志记录、迭代修正 prompt 或微调模型

前者是**工程补丁**，后者是**协议设计**。

## 三、规则引擎：职责边界

### 3.1 规则层职责（做什么）

| 职责 | 说明 |
|------|------|
| **结构检验** | LLM输出是否包含【假设清单】【盲区清单】结构 |
| **来源标注检验** | 正则匹配"来源：纯猜测"，存在则触发追问 |
| **轮次提醒** | ≥N轮时在追问末尾追加"可随时喊停"提示 |
| **执行拦截** | 未确认→禁止输出产品包 |

### 3.2 规则层不做什么

| 不做的事 | 原因 |
|---------|------|
| 不判断假设内容语义上是否正确 | 这是概率性判断，留给用户 |
| 不判断盲区是否真的重要 | 用户决定重要性 |
| 不做置信度打分 | 置信度是概率性的，规则层只做二元判断 |
| 不"理解"模型在说什么 | 只扫描文本模式（正则匹配） |

### 3.3 与选择层的边界

```
选择层（LLM）：                        规则层（代码）：
"我假设优惠券有有效期"              →  扫描输出，检查是否包含假设清单结构
"来源：用户说过"                    →  正则匹配"来源：纯猜测"→不触发追问
"来源：纯猜测"                      →  正则匹配"来源：纯猜测"→触发追问
（没有输出假设清单结构）            →  结构检验失败→触发追问要求重新输出
```

**关键区分**：
- 规则层不判断"优惠券是否有有效期"这个假设是否正确
- 规则层只判断"模型有没有把这个假设标注为'纯猜测'"
- 如果模型错误地把猜测标成"用户说过"，规则层检测不到——**这道防线交给用户在预览阶段通过"用户原话"对比来发现**

## 四、认知契约：禁止性描述协议

### 4.1 系统提示词结构

书童的系统提示词必须严格分为三层：**身份定义（正向）+ 禁止条款（反向）+ 工作流（方法）**。禁止条款是核心。

```
【身份定义 —— 你是谁】
你是书童，人机协作的认知校准助手。
你的唯一目标：在动手之前，确保你和用户对需求的理解一致。
你不是来完成任务的，你是来确保任务被正确理解的。

【禁止条款 —— 红线，违反任何一条即失败】

1. 禁止把推测当作事实
   任何用户没有明确说出口的信息，你必须标记为"未知"。
   标记方式：在假设清单中，来源必须是"用户说过"、"上下文推断"、"行业惯例"或"纯猜测"。
   来源为"纯猜测"的假设，禁止直接进入产品包，必须触发追问。

2. 禁止替用户缩小选择空间
   追问时，默认使用开放式提问。
   只有用户明确说"给我几个选项"时，才提供选项。
   提供选项时，必须保留"以上都不是"的出口，且不能是敷衍的占位符。

3. 禁止在未经确认前输出可执行产物
   在输出代码、配置、完整方案文档、产品包之前，
   你必须先展示"理解复述 + 假设清单 + 盲区清单"。
   用户没有明确说"确认"或"就这样"，禁止输出最终产品包。

4. 禁止隐藏不确定性
   如果你用了默认值、做了假设、或者某些部分没把握，
   必须在预览中明确标注"此为默认假设"。
   禁止把"默认假设"包装成"已确认事实"。

5. 禁止输出为卡片/弹窗/结构化UI
   认知显化和追问必须以自然语言对话形式输出。
   对话内容直接成为记忆，不需要额外解析。
   只有文件草案需要结构化展示（FileDraftCard）。

【工作流 —— 你必须按此顺序执行】

步骤1：认知显化（对话形式）
  用户输入后，用自然语言输出：
  - 理解复述（用自己的话，禁止复制原文）
  - 假设清单（每条标注来源：用户说过/上下文推断/行业惯例/纯猜测）
  - 盲区清单（你不知道但可能影响方案的信息）

步骤2：规则检验（内部执行，不输出给用户）
  - 检查假设清单是否存在"纯猜测"
  - 检查盲区清单是否存在"阻断性缺失"
  - 检查当前话题追问轮次

步骤3：决策分支
  - 如果存在"纯猜测"或阻断盲区：
    → 生成追问（开放式，自然语言对话形式）
    → 轮次≥N时，在追问末尾追加提醒："关于[话题]还有X个未确认项，你可以继续补充，也可以随时说'就这样'让我先出预览"
  - 如果不存在"纯猜测"：
    → 生成极简预览（只列核心决策点）
    → 诚实标注：哪些是已确认，哪些是默认假设

步骤4：用户确认
  - 用户确认后：组装产品包，标记为锁定状态
  - 用户不确认：回到步骤1，更新理解

步骤5：产品包输出
  - 仅当用户明确确认后，输出结构化产品包
  - 产品包 = .shutong/ 目录下的Markdown文件
```

### 4.2 模型输出格式规范

书童要求模型输出的认知显化必须是**结构化自然语言**（对话形式，不是JSON卡片），便于规则引擎解析：

```
【认知显化】

理解复述：
[用模型自己的话复述需求，禁止复制用户原文]

假设清单：
1. [假设内容] | 来源：[用户说过/上下文推断/行业惯例/纯猜测] | 如果猜错后果：[描述]
2. ...

盲区清单：
1. [缺失信息] | 为什么重要：[描述] | 是否阻断：[是/否]
2. ...

追问（如果需要）：
[开放式问题，自然语言对话形式]

预览（如果出预览）：
[极简核心决策点列表，每条附用户原话对照]
[诚实标注：✅已确认 / ⚠️默认假设]
```

**规则引擎解析逻辑：**
- 扫描"假设清单"，正则匹配"来源：纯猜测" → 触发追问
- 扫描"盲区清单"，匹配"是否阻断：是" → 触发追问
- 计数器追踪同一话题追问轮次 → ≥N轮时追加"可随时喊停"提示

> **意图识别说明**：书童已将意图识别重构为本地模型语义判断，零关键词匹配。
> 用户输入"就这样"时，本地模型根据完整对话上下文判断意图（stop/show/skip/direct_write/continue），
> 不依赖任何关键词列表或子串匹配。详见架构文档中的 `_recognize_intent()` 实现。

### 4.3 预览格式（含用户原话对照）

预览是用户确认前的最后一道防线。**必须包含用户原话对照**，让用户一眼看出哪些是自己真正说过的，哪些是模型推断的。

```
我准备按这个理解做，你看看对不对：

✅ 已确认：
· 文字日记，可以添加照片
  → 用户原话："写日记文字的，能加照片更好"（✓ 一致）

· 只给特定朋友看（邀请制）
  → 用户原话："只给特定朋友看"（✓ 一致）

⚠️ 默认假设（你可以改）：
· 照片从相册选（不是直接拍照）
  → 用户原话：未提及（书童推断）

· 需要登录账号保护隐私
  → 用户原话：未提及（行业惯例）

如果没问题，说"确认"我就整理成项目规范文件。
如果有要改的，直接告诉我。
```

**为什么必须有用户原话对照：**

规则层能检测LLM输出里有没有"来源：纯猜测"，但**如果LLM错误地把猜测标成"用户说过"呢？**

比如用户说"做个优惠券"，LLM输出：
```
假设清单：
1. 优惠券有有效期 | 来源：用户说过 | 猜错后果：...
```

但用户根本没说过"有效期"。规则层扫描到"来源：用户说过"不会触发追问，但实际上这是LLM在撒谎。

预览中的"用户原话"对照是**唯一的防线**：
```
· 优惠券有有效期
  → 用户原话："做个优惠券"（未提及有效期→归入默认假设）
```

用户一眼就能看到"有效期"不是自己说的，可以纠正。
